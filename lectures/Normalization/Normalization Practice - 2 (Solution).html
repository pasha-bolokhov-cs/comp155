Normalization Example

Normalize the following table. Be sure to note your normal forms at each stage of the process.

emp no emp name dept no dept name mgr no mrg name proj no proj name start date bill hours
7902 Smith 10 sales 7988 Jones 15 feasibility Sep-01 100
            35 testing Oct-01 100
            45 implement Oct-01 150
7988 Jones 20 marketing 7699 Walker 15 feasibility Aug-01 200
            25 analysis Sep-01 250
            45 implement Oct-01 200
7562 Smith 10 sales 7099 Phillips 25 analysis May-01 150

Solution:

    1. First Normal Form

The primary key is emp_no.

The set of attributes which repeat for each emp_no are proj_no, proj_name, start_date and bill_hours.

Removing these repeating attributes from the full set of attributes leaves the relation:

   employee (emp_no, emp_name, dept_no, dept_name, mgr_no, mgr_name)

The primary key for the repeating group is proj_no because for each emp_no the proj_no uniquely identifies the project details.

   empProjects (emp_no, proj_no, proj_name, start_date, bill_hours)

The first normal form relations are:

    employee( emp_no, emp_name, dept_no, dept_name, mgr_no, mgr_name)

    empProjects( emp_no, proj_no, proj_name, start_date, bill_hours)

    2. Second Normal Form

Must remove all partial dependencies from the relations.  A partial dependency occurs in a relation when a non-key attribute depends on only part of the primary key.

All relations with single attribute primary keys are automatically in second normal form.

Dependencies in the empProjects relation:

    emp_no, proj_no     -->      proj_name, start_date, bill_hours        Primary key determines attributes

    proj_no    -->     proj_name                    The project number determines the project name

Remove the proj_name attribute to a new relation which has a primary key composed of the attribute proj_no.

    project( proj_no, proj_name)

    empProjects( emp_no, proj_no, start_date, bill_hours )

Full list of relations in second normal form:

    project( proj_no, proj_name)

    empProjects( emp_no, proj_no, start_date, bill_hours )

    employee( emp_no, emp_name, dept_no, dept_name, mgr_no, mgr_name)

    3. Third Normal Form

Third normal form relations contain no transitive dependencies.  A transitive dependency occurs when one or more attributes are determined by one or more attributes which are not part of the primary key.

Dependencies in the employee relation:

 1.    emp_no    -->     emp_name, dept_no, dept_name, mgr_no, mgr_name    Primary key determines attributes

 2.   emp_no, dept_no      -->     mgr_no        mgr_no is determined by its department number and the employee number

 3.   mgr_no     -->     mgr_name                    The manager number determines the manager name

 4.   emp_no, dept_no  -->  mgr_name        Transitive dependency from the above (2) and (3)

 5.   dept_no  -->  dept_name            Department number determines department name

 6.  emp_no -->  dept_name            Transitive dependency from the above (1) and (5)

Remove the mgr_name attribute to a new relation which has a primary key composed of the attribute mgr_no.

    manager( mgr_no, mgr_name)

Remove the dept_name attribute to a new relation which has a primary key composed of the attribute dept_no.

    dept( dept_no, dept_name)

The other relations have no transitive dependencies, so the final set of relations in third normal form are:

    employee( emp_no, emp_name, dept_no, mgr_no)

    project( proj_no, proj_name)

    empProjects( emp_no, proj_no, start_date, bill_hours )

    dept( dept_no, dept_name)

    manager( mgr_no, mgr_name)
