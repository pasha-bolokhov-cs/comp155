Normalization Practice - 3

You are given the following table to evaluate for normalization problems.

BRANCH_CUSTOMER(customerNo, branchNo, visitingFreq, dateRelationshipEstablished, salespersonNo)

You are also given the following business rules:

  1. Each branch will serve a customer through only 1 salesperson
  2. Each salesperson works for one branch only

You now need to:

  • Determine if the table is fully normalized
  • If the table is not fully normalized, identify the normalization violation (i.e. give which normal form it violates), and correct the problem.

 

Solution

BRANCH_CUSTOMER(customerNo, branchNo, visitingFreq, dateRelationshipEstablished, salespersonNo)

  • Each branch will serve a customer through only 1 salesperson

  • Each salesperson works for one branch only

salespersonNo  --> branchNo        Branch is determined by salesperson

 But if we create a table

        SALESPERSON(salespersonNo, branchNo)

then we can’t make changes to the original table because branchNo is part of the key so we can’t remove it.

Note that even though salespersonNo is a determinate, it is not a candidate key on its own. In other words, we can’t get rid of the key customerNo, branchNo  and replace it with salespersonNo. This means that we have violated Boyce-Codd normal form. To be in BCNF, every determinate must be a candidate key.

But have another candidate key:  customerNo, salespersonNo

 BRANCH_CUSTOMER(customerNo, salespersonNo, visitingFreq, dateRelationshipEstablished, branchNo)

 But that relation is not even in 2NF

BRANCH_CUSTOMER(customerNo, salespersonNo, visitingFreq, dateRelationshipEstablished)

 SALESPERSON(salespersonNo, branchNo)

 

But if dateRelationshipEstablished is the date that the BRANCH established a relationship with the customer

CustomerNo + branchNo dateRelationshipEstablished

CUSTOMER_BRANCH(customerNo, branchNo, dateRelationshipEstablished)
